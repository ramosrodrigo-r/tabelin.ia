## Cross-Phase Integration

A wiring entre **12 → 13 → 14** está completa: os 21 REQ-IDs em escopo (UNI, CLAR, TAB, LOC, SEC-05) participam da cadeia `route ↔ schema union ↔ render-dispatcher ↔ componentes`. Typecheck limpo; 105 testes verdes em 6 arquivos.

**E2E flows completos (4):**
1. Input unificado → classificação de intent → 7 resolvers legados (sem regressão).
2. Pedido de tabela → loop de clarificação → ClarificationCard → teto MAX_CLAR_TURNS=2 → geração.
3. `table_spec` com rows → TableGridPanel → useFormulaEngine → recálculo vivo → formatação BR → SEC-05.
4. Quota: `reserveToolUse` único; clarification path `releaseToolUse` (sem débito), generation path `confirmToolUse` (débito único).

**Achados não-bloqueadores (2 — tech debt):**
- **Rota órfã** `DELETE /api/conversations/unified` — `handleNewConversation` só reseta estado client; "nova conversa" não limpa exchanges `unified_table` no banco → pode contar clarTurns de conversa anterior e disparar o teto **CLAR-02** prematuramente.
- **Round-trip frágil do ConfirmationCard** quando `table_spec.rows` está ausente — `resolveOverrideSpec` aceita spec sem rows e o dispatcher reroteia para ConfirmationCard (loop potencial). Provavelmente inalcançável em produção (fixture sempre tem rows; prompt exige rows), mas sem guard defensivo. Afeta **CLAR-04**.
