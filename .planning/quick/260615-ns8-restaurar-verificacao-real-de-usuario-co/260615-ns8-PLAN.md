# Plan: Restaurar verificação real de usuário (Better Auth)

## Context

`apps/web/src/app/api/auth/[...all]/route.ts` reimplementa auth na mão. O
`catch` de `persistCredentials` (linhas 74-81) trata QUALQUER erro do Prisma
(ex.: DB fora do ar) como sucesso, em DEV retornando
`{ ok: true, userId: undefined, name }` tanto para sign-up quanto para
sign-in — e o handler então emite um cookie de sessão válido via
`createSessionToken`/`createSessionUser`. Resultado: em dev, qualquer
email/senha em `/api/auth/sign-in/email` entra no `/workspace` sem conta
existente.

Já existe uma instância real do Better Auth em
`apps/web/src/server/auth/config.ts` (prismaAdapter + emailAndPassword +
secret + trustedOrigins) e o schema Prisma já tem os modelos
User/Session/Account/Verification no formato esperado pelo Better Auth — mas
nada disso está conectado.

## Estratégia

Trocar o handler manual pelo handler oficial do Better Auth
(`toNextJsHandler(auth.handler)`), e reimplementar `getCurrentUser()` /
`getSessionFromCookieHeader()` em `apps/web/src/server/auth/session.ts` por
cima de `auth.api.getSession({ headers })`, **preservando as assinaturas e o
tipo `SessionUser`** já consumidos por 6 arquivos
(`app/page.tsx`, `app/(workspace)/workspace/layout.tsx` via
`request-cache.ts`, `components/app/topbar.tsx`,
`api/chat/unified/route.ts`, `api/workspace/state/route.ts`,
`api/conversations/unified/route.ts`, `api/workspace/import/route.ts`).
Isso elimina o blast radius — esses arquivos não precisam mudar.

Better Auth, com `emailAndPassword.enabled: true`, já fornece nativamente:
`/api/auth/sign-up/email`, `/api/auth/sign-in/email`, `/api/auth/sign-out`,
`/api/auth/forget-password`, `/api/auth/reset-password`, `/api/auth/get-session`
— todos os paths usados pelas páginas em `app/(auth)/`. O `trustedOrigins`
já configurado cobre a validação de origem (Better Auth aplica
`originCheck` em endpoints state-changing), então o
`validateAuthPostOrigin` manual pode ser removido.

## Tasks

### Task 1: Substituir o handler manual pelo handler oficial do Better Auth

**Arquivo:** `apps/web/src/app/api/auth/[...all]/route.ts`

Reescrever o arquivo inteiro para:

```ts
import { toNextJsHandler } from "better-auth/next-js";

import { auth } from "@/server/auth/config";

export const { GET, POST } = toNextJsHandler(auth.handler);
```

Remover todo o código anterior deste arquivo: `persistCredentials`,
`getOrigin`/`getRequestSourceOrigin`/`getAllowedAuthOrigins`/
`validateAuthPostOrigin`, `getSafeResetPath`/`buildResetUrl`/
`createResetLinkIfUserExists`, e os imports de `@/server/auth/session`,
`@/server/auth/password`, `@/server/auth/reset-password`, `@/server/db/client`.

**Verificação:** após o Task 3, `npm run typecheck` (workspace `apps/web`)
não deve apontar para este arquivo.

### Task 2: Reimplementar `getCurrentUser` / `getSessionFromCookieHeader` sobre `auth.api.getSession`

**Arquivo:** `apps/web/src/server/auth/session.ts`

Reescrever o arquivo para manter apenas o que ainda é necessário:

- Tipo `SessionUser` (`{ id, email, name }`) — mantido para não quebrar
  `Topbar`, `layout.test.tsx`, etc.
- `getCurrentUser()`: usa `auth.api.getSession({ headers: await headers() })`
  (import `headers` de `next/headers`), mapeia `session.user` →
  `SessionUser` (name pode ser `null` no schema — fazer
  `name: session.user.name ?? session.user.email.split("@")[0]`). Retorna
  `null` se não houver sessão.
- `getSessionFromCookieHeader(cookieHeader: string | null)`: usado pelas API
  routes que recebem `Request` puro (não Next `headers()`), portanto
  construir um `Headers` a partir do cookie header
  (`new Headers(cookieHeader ? { cookie: cookieHeader } : {})`) e chamar
  `auth.api.getSession({ headers })` (síncrono → função precisa virar
  `async`; ajustar os 4 call sites — eles já estão em handlers `async`, só
  trocar para `await getSessionFromCookieHeader(...)`).
- Remover: `SESSION_COOKIE`, `createSessionUser`, `createSessionToken`,
  `verifySessionToken`, `normalizeEmail`, toda a lógica HMAC/base64url
  (`createHmac`, `timingSafeEqual`, etc.) — não são mais usados por
  ninguém após o Task 1.

Importar `auth` de `@/server/auth/config`.

**Ajustar 4 call sites** (tornar `await`):
- `apps/web/src/app/api/chat/unified/route.ts:275`
- `apps/web/src/app/api/workspace/state/route.ts:15`
- `apps/web/src/app/api/conversations/unified/route.ts:13`
- `apps/web/src/app/api/workspace/import/route.ts:66`

**Verificação:** `grep -rn "SESSION_COOKIE\|createSessionToken\|createSessionUser\|verifySessionToken\|normalizeEmail" apps/web/src` retorna vazio.

## Notas / Riscos

- Better Auth nomeia o cookie de sessão diferente do
  `tabelin_session` antigo (default `better-auth.session_token`, prefixo
  configurável). Sessões antigas (cookie HMAC custom) ficam inválidas — é
  esperado, usuários precisarão logar de novo. Não é necessário código de
  migração.
- `BETTER_AUTH_SECRET` já está configurado em `config.ts`; confirmar que a
  env var existe no `.env` local (Better Auth lança erro se ausente em
  alguns modos — verificar durante o `npm run dev` do Task 3.4).
- Não adicionar providers OAuth, verificação de email adicional, nem outros
  endpoints — escopo é só restaurar o caminho credential (email+senha) que
  já estava em uso.
