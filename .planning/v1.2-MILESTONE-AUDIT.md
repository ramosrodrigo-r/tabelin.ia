## Integração Cross-Fase

**Veredito do integration checker:** PASS com um warning. A cadeia completa está costurada end-to-end e aplicada de forma **idêntica nos 5 tools**: UI hook → FormData → route multipart → `extractContent` → `attachmentContext` → `buildToolContextMessages` → injeção no system prompt → persistência → emit `attachment_grounded` → captura no hook → grounding badge / painel de transparência. Nomes de campo batem em `packages/shared` ↔ routes ↔ hooks, sem drift.

### Seams verificados

| Seam | Direção | Status |
|------|---------|--------|
| UI hook → route | 11 → 10 | WIRED — FormData quando há file, JSON caso contrário; `attachment_grounded` shape idêntico nos 5 |
| route → extraction | 10 → 09 | WIRED — todos chamam `extractContent`; ambos branches (ok/erro) tratados; quota liberada em toda falha |
| extraction → LLM context | 09 → 10 | WIRED — `result.text` → `attachmentContext` → 5º arg → injeção delimitada → truncado a 8000 → persistido |
| Pro-gate consistency | 10/11 | WIRED — 403 antes da extração nos 5 routes (template incondicional preservado); botão disabled para free nos 5 |
| SEC-01 (render seguro) | 11 | WIRED — zero `dangerouslySetInnerHTML`; `<pre>{extractedText}</pre>` |
| **Erro de extração → UI** | **10 → 11** | **BROKEN (warning)** — hooks não surfacam 422/413 |

### ⚠ Warning de integração (não-blocker)

**SEAM-05 — mensagens de erro de extração não chegam ao usuário** (afeta EXT-03, EXT-06, ATT-04):
Os routes retornam falhas de extração como HTTP **422** com `result.message` acionável em pt-BR (ex.: PDF escaneado → *"Este PDF parece ser escaneado... Use o tool de OCR..."*) e **413** para `FILE_TOO_LARGE`. Nenhum dos 5 hooks (`use-{tool}-stream.ts`) tem branch para esses status — caem no `setError` genérico *"Nao consegui validar a resposta."*. A orientação acionável que o backend produziu (SCANNED_PDF, UNSUPPORTED_TYPE, EMPTY_EXTRACTION, FILE_TOO_LARGE) é descartada antes de chegar ao usuário.

**Por que não é blocker:** EXT-06 foi satisfeito na camada de extração (Phase 09 — o backend produz o erro acionável corretamente, com teste verde). A lacuna é apenas a propagação para a UI — uma degradação de UX, não uma quebra de fluxo. O usuário ainda é bloqueado/avisado, só com mensagem menos específica.

**Fix sugerido:** adicionar branch `response.status === 422 || response.status === 413` lendo `errorData.message` no bloco `!response.ok` dos 5 hooks em `apps/web/src/features/{formula,sql,regex,scripts,template}/hooks/use-{tool}-stream.ts`.
