<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL pathname -> Topbar toolKind | Client route determines which conversation delete endpoint is called |
| Topbar -> conversation delete API | Browser action deletes persisted user history |
| Root workspace -> feature component | Server component decides default workspace experience |
| Conversation delete route -> database | Authenticated route deletes rows by userId and toolKind |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-04-01 | Tampering | Conversation delete tool kind | mitigate | Existing `[tool]` route validates against `VALID_TOOL_KINDS`; unified route uses hardcoded server-side `ALL_UNIFIED_TOOL_KINDS`. |
| T-12-04-02 | Information Disclosure | Cross-user history deletion | mitigate | Both delete routes derive user id from signed session cookie; no userId is accepted from the client. |
| T-12-04-03 | Denial of Service | Unified delete fanout | accept | Six bounded delete calls only; no user-controlled list or unbounded loop. |
| T-12-04-04 | Repudiation | Destructive new conversation action | mitigate | Existing confirmation popover remains; unified copy explicitly says the whole unified chat history is deleted. |
| T-12-04-05 | Regression | Per-tool access | mitigate | Tests and source checks keep `/workspace/sql`, `/workspace/regex`, `/workspace/scripts`, and `/workspace/templates` mapping intact. |
</threat_model>
