<objective>
Fechar os dois gaps bloqueadores de segurança (CR-01 e CR-02) que impedem SEC-02 de ser declarado satisfeito.

CR-01: `guardXlsxZip` confia em `info.originalSize` (metadado do central directory, gravável pelo atacante) sem verificar a proporção comprimido/descomprimido nem um cap por-entrada. Um ZIP bomb que declare `originalSize=1` passa o guard e explode em `XLSX.read`. A defesa real é o ratio check (originalSize / info.size — comprimido) que ainda não existe.

CR-02: `extractContent` aceita Buffer de tamanho arbitrário e faz dupla alocação (~2N bytes) com `new Uint8Array(buffer)` + `new ArrayBuffer(buffer.length)` antes de qualquer verificação. Não existe `MAX_INPUT_BYTES` guard.

Purpose: Tornar o sistema de extração resistente ao ataque real de ZIP bomb com metadado forjado e ao ataque de exaustão de memória por input de tamanho arbitrário — ambos requisitos de SEC-02/D-11.
Output: `zip-guard.ts` com ratio cap + per-entry cap; `dispatcher.ts` com input size guard; `types.ts` com `FILE_TOO_LARGE`; testes de segurança estendidos cobrindo os dois vetores.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente HTTP → extractContent | Ponto de entrada de bytes não confiáveis. Um atacante pode enviar arquivos de tamanho arbitrário ou ZIPs com metadados do central directory forjados. |
| ZIP central directory → guardXlsxZip | Os campos info.originalSize do central directory são escritos pelo criador do ZIP — um atacante os controla livremente. info.size (comprimido) reflete bytes reais no arquivo e é mais confiável para ratio checks. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-09-05-01 | Tampering | zip-guard.ts — info.originalSize | mitigate | Adicionar ratio check (info.originalSize / info.size > MAX_RATIO=100): mesmo que o atacante declare originalSize pequeno, um deflate stream real produz ratio real alto que é capturado. info.size (comprimido) é o campo usado para a defesa real (CR-01). |
| T-09-05-02 | Denial of Service | zip-guard.ts — per-entry size | mitigate | Adicionar MAX_ENTRY_UNCOMPRESSED=25 MB: qualquer entrada que declare originalSize acima de 25 MB é rejeitada imediatamente, sem descompactar. Cap por-entrada complementa o cap total (MAX_TOTAL_UNCOMPRESSED). |
