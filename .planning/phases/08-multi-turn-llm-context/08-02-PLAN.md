<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| histórico → array de mensagens do LLM | streams replayam turnos anteriores na chamada real ao modelo |
| toolKind por stream → isolamento de thread | o toolKind passado ao helper determina qual thread é injetado |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-04 | Information Disclosure (vazamento cross-tool) | toolKind literal em cada stream | mitigate | Cada stream passa seu próprio toolKind literal (`"sql"`/`"regex"`/`"script"`/`"template"`) — scripts usa `"script"` singular para bater com o save existente. O isolamento real vem do filtro no route (Plano 03); aqui garantimos que o literal não cruza tools. Coberto por asserção de string no acceptance_criteria. |
| T-08-05 | Tampering (explain recebendo contexto) | branch explain do regex-stream | mitigate | D-03: branch `explain` permanece intocado e sem history; asserção no acceptance_criteria verifica que explain não referencia `buildToolContextMessages`. |
| T-08-06 | Denial of Service (token overflow) | injeção no caminho real do LLM | mitigate | `truncateHistory` aplicado antes de montar as mensagens em todos os 4 streams (herdado do Plano 01, MULTI-02). |
</threat_model>
