<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → route handler | Request HTTP — body validado com zod antes de qualquer operação |
| route handler → saveConversationExchange() | userId vem de getSessionFromCookieHeader() (autenticado), nunca do body do request |
| route handler → response | Exchange salvo antes do return — falha de persistência não bloqueia a resposta |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Spoofing (auth bypass) | Todos os 7 route handlers | accept | getSessionFromCookieHeader() já aplicado em todos os handlers antes de qualquer operação. userId vem exclusivamente da sessão autenticada — nunca do body do request. Guard existente, não modificado. |
| T-06-06 | Elevation of Privilege (IDOR) | saveConversationExchange() — userId parameter | mitigate | userId passado ao repository vem de user.id (sessão autenticada), não de parsed.data. Um usuário não pode salvar exchange em nome de outro. Verificar em code review que nenhum handler passa userId do body. |
| T-06-07 | Tampering (save antes de confirmToolUse) | Todos os 7 route handlers — ordem de chamadas | mitigate | saveConversationExchange() inserida APÓS confirmToolUse() e recordToolRequest() em todos os handlers. Exchanges só são persistidos para gerações bem-sucedidas — nenhum registro com payload corrompido (per Pitfall 3 do RESEARCH.md). |
| T-06-08 | Availability (falha de persistência quebra resposta) | conversation-repository.ts — try/catch | accept | try/catch/warn silencioso no repository — falha de banco de histórico retorna null e emite console.warn sem relançar exceção. Resposta ao usuário não é afetada por falha de persistência. |
| T-06-SC | Tampering | npm/pip/cargo installs | accept | Nenhum pacote novo instalado nesta fase. Todos os imports usam módulos existentes do projeto. |
</threat_model>
