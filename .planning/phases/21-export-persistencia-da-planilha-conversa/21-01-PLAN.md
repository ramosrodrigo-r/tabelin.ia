# Phase 21, Plan 1: Banco de Dados, API Endpoint & Workspace State Provider

Este plano cobre o setup do banco de dados (funções helper em `conversation-repository.ts`), a criação da rota de API `/api/workspace/state` e a atualização do `WorkspaceStateProvider` para suportar inicialização e auto-save debancado.

## Tasks

### Task 1: Adicionar Helpers no Repositório de Conversas
Adicionar as funções canônicas para salvar e ler o spec da planilha e buscar o histórico unificado de conversas no Prisma.

- **File**: `apps/web/src/server/tools/conversation-repository.ts`
- **Changes**:
  - Implementar `getActiveSpreadsheetSpec(userId: string): Promise<TableSpecPayload | null>` que busca o registro mais recente com `toolKind: "unified_table"`.
  - Implementar `saveActiveSpreadsheetSpec(userId: string, spec: TableSpecPayload): Promise<void>` que deleta registros antigos `"unified_table"` e cria o novo registro em uma transaction Prisma.
  - Implementar `findUnifiedConversationExchanges(userId: string): Promise<ConversationExchange[]>` que busca os registros com `toolKind` em `["sheet_operation", "qa"]` e `mode: GENERATE_MODE`, ordenados por `createdAt: "asc"`.

### Task 2: Criar Rota do API de Estado da Planilha
Criar o endpoint `/api/workspace/state` que recebe e valida o payload da planilha e chama a gravação no repositório.

- **File**: `apps/web/src/app/api/workspace/state/route.ts`
- **Changes**:
  - Implementar o método `POST` que valida a sessão do usuário via `getSessionFromCookieHeader`.
  - Parsear e validar o corpo da requisição usando `tableSpecPayloadSchema`.
  - Chamar `saveActiveSpreadsheetSpec` e retornar status `200` com `{ ok: true }` ou retornar erros de validação/autenticação adequados.

### Task 3: Atualizar WorkspaceStateProvider
Modificar o provedor de estado no frontend para aceitar a planilha inicial e efetuar o auto-save debancado.

- **File**: `apps/web/src/components/app/workspace-state-context.tsx`
- **Changes**:
  - Atualizar `WorkspaceStateProvider` para aceitar a prop opcional `initialSpec?: TableSpecPayload`.
  - Mudar `initialPresent` para inicializar a partir de `initialSpec` se fornecido (usando `seedToGridState(initialSpec)`), caindo de volta em `SAMPLE_SPEC`.
  - Criar um `lastSavedRef` inicializado com a representação em string do estado inicial.
  - Adicionar um `useEffect` com dependência em `history.present` e `spec` que compare o estado atual com `lastSavedRef.current` e, caso sejam diferentes, dispare um POST para `/api/workspace/state` debancado por 1.5 segundos.
  - No sucesso do POST, atualizar o `lastSavedRef.current` para evitar envios redundantes.
